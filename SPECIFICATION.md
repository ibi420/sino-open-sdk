# Sino Cryptographic Specification & Architecture

This document provides the formal open specification for **Sino's Zero-Knowledge Encryption Layer**.

---

## 1. Cryptographic Key Hierarchy

Sino enforces a 4-tier cryptographic key hierarchy to ensure local encryption and RAM isolation:

```
[ User Password / PIN ]
          │
          ▼  (Argon2id - 64MB RAM, 3 iterations, 4 parallelism)
[ Password Hash / Key ]
          │
          ▼  (AES-256-GCM Unwrap)
[ Root Master Key (RMK) ]
          │
          ▼  (AES-256-GCM Key Wrapping)
[ Data Encryption Key (DEK) ] ──▶ Encrypts Payload Bytes
```

---

## 2. Encryption Engine Specification

- **Algorithm**: AES-256-GCM (`AES/GCM/NoPadding`)
- **Key Length**: 256 bits (32 bytes)
- **IV Length**: 96 bits (12 bytes)
- **Authentication Tag Length**: 128 bits (16 bytes)

### 2.1 Chunked GCM Streaming Specification (Age-Style)
For large files and direct cloud media streaming:
1. Payloads are divided into **1MB fixed chunks** (1,048,576 bytes).
2. Each chunk is encrypted with a distinct IV generated by incrementing the last 4 bytes of the base IV:
   $$\text{IV}_{\text{chunk}} = \text{BaseIV} \oplus \text{BigEndian32}(\text{ChunkIndex})$$
3. Each 1MB chunk carries its own 16-byte GCM authentication tag, enabling random-access range requests (`HTTP 206`) directly from cloud storage without downloading unneeded bytes.

---

## 3. Cloud Blindness & Path Anonymization

Sino converts human-readable relative file paths (`DCIM/Vacation/photo.jpg`) into opaque, un-linkable cloud object keys using **HMAC-SHA256**:

$$\text{CloudKey} = \text{HMAC-SHA256}(\text{Salt}_{\text{path}}, \text{NormalizedPath})$$

Cloud providers see only opaque 64-character hexadecimal keys and encrypted payloads, remaining completely blind to:
1. Original file names & extensions.
2. Directory structure & nesting hierarchy.
3. Original file size (due to compression & 1MB chunk alignment).

---

## 4. Dual-Vault Duress Isolation Specification

- Primary Vault and Decoy (Duress) Vault use **domain-separated Argon2id salts**:
  - `SinoPrimaryMasterKeySaltV1` for Primary Vault
  - `SinoDuressDecoyKeySaltV1` for Decoy Vault
- Primary keys and Decoy keys share zero mathematical linkage. Analyzing the algorithm or code structure gives zero cryptographic indicator of whether a secondary decoy vault exists.

---

## 5. Memory Hygiene Specification

All transient sensitive byte arrays (`ByteArray` representing DEK, RMK, or plaintexts) are explicitly zeroed out in `finally` blocks using `SecurityUtils.fillZero(array)`:

```kotlin
try {
    // Encrypt / Decrypt payload
} finally {
    SecurityUtils.fillZero(dekBuffer)
}
```
